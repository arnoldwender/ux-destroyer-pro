# Contributing to UX Destroyer Pro

Welcome, destroyer. You've entered the anti-pattern laboratory where every UX principle you've ever learned goes to suffer. Nielsen would not approve. That's the point.

## Getting Started

```bash
git clone https://github.com/arnoldwender/ux-destroyer-pro.git
cd ux-destroyer-pro
npm install
npm run dev
```

Open [http://localhost:5173](http://localhost:5173). Toggle some anti-patterns on. Try to survive Boss Mode. Report back (if you can find the back button).

## Submitting a Pull Request

1. Fork the repo and create your branch from `main`
2. Make your changes
3. Make sure the app still builds (`npm run build`)
4. Open a PR describing your contribution. Use a real close button on the PR, not a fake one.
5. If you're adding a new anti-pattern, explain which UX principle it violates and why that's funny

## Code Style

- **TypeScript** — the code itself must be clean, even if the output is a crime against usability
- **React** functional components with hooks
- Anti-patterns should be toggleable and isolated. No anti-pattern should break the app's core functionality when enabled.
- There's a fine line between "funny bad UX" and "actually inaccessible." Stay on the funny side.

## Bug Reports

Open an issue with:
- What you expected to happen
- What actually happened
- Your browser and OS
- Which anti-patterns were enabled (important — some combinations are known to cause existential dread)

## Feature Requests

New anti-patterns, dark pattern showcases, or survival challenges? Open an issue! Study real dark patterns in the wild for inspiration — the best satire comes from reality.

## The Golden Rule

We destroy UX so others don't have to. Every anti-pattern in this app is a lesson in what not to do. Keep the satire sharp and the underlying code professional.
